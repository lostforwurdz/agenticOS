# Full Audit Weekly Agent

**Schedule:** Sundays 10:00 America/Chicago (`0 10 * * 0`)
**Scope:** Primary active project. Hardcoded to `loom`; later, read from `bd memories "agenticos.primary-project"`.
**Venue:** Cloud routine (RemoteTrigger).
**Precondition:** Requires `/web-setup` GitHub connection for `lostforwurdz/loom` (else the routine can't push the audit branch or open a PR).

## Goal

Run the full 9-agent parallel audit on the primary repo weekly, surface findings as a PR-attached FIXES.md, and ensure critical issues are tracked in bd before the next work session begins.

## Steps

1. **Resolve primary project**

   Use the hardcoded value `loom` as the primary project repo. The repo must be present as a cloned git repository in the routine environment. If the repo is unavailable, abort immediately and write a failure note to bd:
   ```bash
   bd remember --key "full-audit.loom.$(date +%F)" "ABORTED: repo not available in routine environment"
   ```

2. **Invoke `/full-audit` workflow**

   Run the `/full-audit` workflow against the `loom` repo. This dispatches 9 specialist agents in parallel (security-auditor, bug-auditor, perf-auditor, compliance-auditor, accessibility-tester, dependency-manager, docs-manager, api-designer, ui-auditor) and consolidates their output into a single `FIXES.md` file. Capture the full output including the consolidated FIXES.md content.

3. **Commit FIXES.md to a dated branch**

   Compute `TODAY` as the ISO date (`YYYY-MM-DD`) at run time. Create and push a branch:
   ```bash
   git checkout -b audit/<TODAY>
   # Write FIXES.md content from the workflow output
   git add FIXES.md
   git commit -m "chore: weekly full-audit FIXES.md <TODAY> [automated]"
   git push origin audit/<TODAY>
   ```
   Only FIXES.md is committed — no source code changes, no auto-fixes.

4. **Open a PR**

   Open a pull request from `audit/<TODAY>` into `main`:
   - Title: `weekly full-audit <TODAY>`
   - Body: paste the executive summary section from FIXES.md plus a note that this was generated by the full-audit-weekly scheduled agent
   - Do not assign reviewers or add labels automatically

5. **File P0 bd issues for critical findings**

   Parse FIXES.md for findings marked as Critical (severity = Critical or P0). For each:
   ```bash
   bd create --priority=0 --type=bug \
     --title="full-audit: loom <finding-title>" \
     --description="<finding-details>\n\nSource: PR audit/<TODAY>"
   ```
   One issue per distinct critical finding. Include the PR branch reference in the description.

6. **Write summary to bd**

   ```bash
   bd remember --key "full-audit.loom.<TODAY>" "<summary>"
   ```
   Summary must include: run timestamp, total findings by severity (Critical/High/Medium/Low), PR URL, and count of P0 issues filed.

## Constraints

- Never commit anything other than FIXES.md — no source edits, no auto-applied patches
- Never merge the PR — it is opened for human review only
- Never force-push
- Total runtime cap: 60 minutes; if the `/full-audit` workflow does not complete within 55 minutes, abort, write the partial output to a bd memory key, and note the timeout in the PR body if the branch was already pushed
- One PR per week maximum — if a branch `audit/<TODAY>` already exists, skip branch creation and reuse the existing branch

## Verification

A successful run produces an open PR in the `loom` repo titled `weekly full-audit <TODAY>`. Confirm the bd memory key is written:
```bash
bd memories "full-audit.loom.$(date +%F)"
```
Output must be non-empty and must include a PR URL.
